# End-of-Life

Very few repositories plan for retirement. We do. A product is retired responsibly — customers are
protected and knowledge is preserved.

## Retire a product only after

- **Migration path exists** — customers can move to an alternative.
- **Customers informed** — with adequate notice and clear timelines.
- **Documentation updated** — migration guides published.
- **Support completed** — the final support window has run.
- **Infrastructure removed** — resources decommissioned to stop cost.
- **Legal obligations satisfied** — data-retention and contractual duties met.
- **Historical data archived** — customer data exportable; records archived.

## Process

1. Decision recorded ([decision memory](../memory/decision-memory.md)) via [product evolution](product-evolution.md).
2. [Deprecation](deprecation-policy.md) announced with timeline.
3. Data export offered; migration guides published.
4. Support and security windows run to their end.
5. Infrastructure shut down; costs stopped.
6. Knowledge transferred; [project memory](../memory/project-memory.md) and repository archived (read-only).

Retirement is a dignified close, not an abandonment.
